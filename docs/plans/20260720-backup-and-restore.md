# Бекап і відновлення даних (знімок + недеструктивний імпорт)

## Overview
- Додати на сторінку *Налаштування* дві можливості (admin-only):
  - **Бекап** — завантажити повний знімок даних одним `.zip` (файл SQLite +
    тека `uploads/` + `manifest.json`). Лише читання, нічого не змінює.
  - **Відновлення** — завантажити такий `.zip` і **недеструктивно імпортувати**
    його в живу БД: додаються лише рядки, яких ще немає (за бізнес-ключами), з
    ремапом зовнішніх ключів. Наявні дані не змінюються й не видаляються.
- Проблема, яку розв'язує: зараз бекап можливий лише вручну через SSH (`tar data/`
  у `docs/deploy.md`). Ця фіча дає це з UI, без доступу до шелу/Synology.
- **Важливі рішення (узгоджено 2026-07-20):** без перезапуску контейнера; повністю
  недеструктивно; відновлення — це **імпорт-злиття** (не відкат/заміна). Планові
  бекапи, вибірковий імпорт і hot-swap відкат — поза обсягом.

  bind-mount `/data`. Тести — лише в Docker (`docker-compose.dev.yml`).

## Solution Overview
- **Бекап (`services/backup.py`):** консистентний знімок БД через
  `VACUUM INTO <tmp>` (не сире копіювання, бо запис іде в тому ж процесі); зібрати
  `.zip` із `hometrap.db`, деревом `uploads/`, `manifest.json`
  (`{app_version, alembic_revision, created_at, db_sha256}`). `VACUUM INTO`
  виконується окремим **raw-з'єднанням** до `database_path` (не через
  request-`Session`, бо VACUUM не працює в транзакції); ревізія читається з
  `alembic_version` тієї ж БД. Стрім як завантаження, temp прибирається.
- **Відновлення (`services/restore.py`):** **безпечно** розпакувати архів у temp
  (guard від zip-slip: відхиляти абсолютні/`..` імена членів); провалідувати
  структуру, `manifest` (ревізія == поточній — міжсхемне злиття небезпечне) і
  **sha256** витягнутого `hometrap.db` проти `db_sha256`; **відхилити** несумісне/
  пошкоджене/ворожеe. Відкрити БД бекапу окремим engine (RO). У
  **одній транзакції** живої сесії пройти сутності в порядку залежностей і зробити
  **upsert-missing за бізнес-ключем** із ремапом FK на знайдений-або-новий батько;
  скопіювати відсутні файли вкладень. Помилка → повний rollback (атомарно,
  недеструктивно). Повернути зведення (додано/пропущено по сутностях).
- **API (`routers/settings.py`):** `GET /api/settings/backup` (завантаження),
  `POST /api/settings/restore` (multipart). Обидва — лише автентифікований admin.
- **Frontend:** секція «Бекап і відновлення» у *Налаштуваннях*: кнопка завантаження;
  завантаження файлу + підтвердження + показ зведення/помилок.
- **Доступ:** наявна залежність `require_auth` (автентифікований користувач);
  фактично це єдиний seeded-admin, бо в моделі `User` немає ролей.
- **Безпека:** архів містить приватні дані (нотифікаційні креди в `Setting`,
  файли договорів) — temp прибирається, вміст не логуємо, не в публічні шляхи.

## Technical Details

### Manifest (`manifest.json`)
```
{ "app_version": "0.1.0", "alembic_revision": "<version_num>",
  "created_at": "<ISO-8601 UTC>", "db_sha256": "<hex>" }
```

### Бізнес-ключі для upsert-missing (порядок залежностей)
- **Apartment** — `(name, address)`
- **Service** — `(apartment*, name)`
- **Tariff** — `(service*, valid_from)`  *(є `uq_tariffs_service_date`)*
- **Tenant** — `(apartment*, full_name, contract_start)`. ⚠️ Через частковий
  унікальний індекс `uq_tenants_active_apartment` (один активний на квартиру,
  `contract_end IS NULL`): якщо квартира вже має ІНШОГО активного орендаря —
  імпортований активний орендар **пропускається** й потрапляє у зведення (щоб не
  зривати транзакцію)
- **TenantAttachment** — `(tenant*, stored_name)`; зливається для НОВИХ і вже
  наявних (matched) орендарів + копіювання відсутнього файлу
- **Invoice** — `(apartment*, period)`  *(є `uq_invoices_apartment_period`)*
- **InvoiceLine** — імпортується лише разом із НОВИМ Invoice (одиниця злиття —
  рахунок); якщо рахунок уже існує — його рядки не чіпаємо
- **ExchangeRate** — `(date, currency)`  *(є `uq_exchange_rates_date_currency`)*
- **User, Setting, PushSubscription** — **не імпортуємо** (логін/налаштування не
  затираємо; push-підписки прив'язані до пристрою й мають унікальний endpoint)

`*` — FK ремапиться на існуючий (якщо батько знайдений за ключем) або на щойно
створений запис.

### Правила
- Рядок, що збігся за ключем, але має інший вміст — **пропускаємо** (не
  перезаписуємо: недеструктивно). Ідемпотентність: повторний імпорт того самого
  архіву не додає нічого.
- Перед вставкою — перевірка не лише за бізнес-ключем, а й за унікальними
  обмеженнями БД (напр. активний орендар); конфлікт → пропуск + запис у зведення,
  **без зриву транзакції**.
- Порядок для файлів: `flush` (отримати нові tenant id) → скопіювати відсутні файли
  вкладень → `commit`. Помилка → прибрати щойно скопійовані файли, потім `rollback`
  (недеструктивно й на рівні ФС; без орфанних файлів).
- Транзакція на весь імпорт; помилка → rollback, жодних часткових змін.

## Post-Completion
*Потребує ручних дій — без чекбоксів*

**Ручна перевірка (прод):**
- завантажити бекап із прода; імпортувати його в локальний Docker → дані сходяться
- імпорт у БД із наявними даними → дублів немає, наявне не змінилось
- перевірити, що під час імпорту сервіс не перезапускається

**Security-огляд:**
- ендпоінти лише для admin; temp-файли прибираються; вміст архіву не логується
- архів містить секрети (нотифікаційні креди) і приватні файли — попередити
  користувача поводитися з ним як із секретом (нагадування в UI/доках)

**Деплой:**
- зміна зачіпає backend+frontend → rebuild за `docs/deploy.md`
  (`up -d --build`). Автодеплой не виконується.
