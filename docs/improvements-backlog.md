# Беклог покращень і нових можливостей HomeTrap

Живий документ із ідеями для підвищення юзабіліті порталу. Джерело — спільний
брейнсторм на основі аналізу коду HomeTrap та огляду ринку property-management
порталів (TenantCloud, DoorLoop, Buildium, RentRedi, Baselane).

## Як користуємось

1. **Уточнення.** Проходимо ідеї одна за одною, уточнюємо обсяг, заповнюємо блоки
   «Уточнення» та «Критерії готовності», ставимо статус `✅ уточнено`.
2. **Оновлення.** Після уточнення список може мінятись: пункти зливаються,
   діляться, переставляються за пріоритетом, об'єднуються в епіки.
3. **Реалізація.** Беремо **по одній ідеї (або одній складовій епіку) за цикл** (щоб
   уникати плутанини): уточнена ідея → окремий план (`docs/plans/`) → імплементація
   → перевірка → коміт. Наступну починаємо лише після завершення попередньої.

## Легенда

- **Тип:** 🔧 покращення наявного · ✨ нове
- **Вплив:** 🔴 високий · 🟡 середній
- **Обсяг:** S (кілька днів) · M (1–2 тижні) · L (3+ тижнів) — орієнтовно
- **Статус:** 📥 беклог · 🔧 уточнюється · ✅ уточнено · 🔽 знижено · 🔗 об'єднано · 🏗️ реалізується · ✔️ готово
- **Номери** `#N` — стабільні ідентифікатори; пріоритет задає хвиля (ідея може змінити хвилю, зберігаючи номер).
- **Епіки** — великі ідеї з підскладовими (T0, T1…); реалізуються складова за складовою.

## Що вже реалізовано (не дублюємо)

Автозаповнення попередніх показань лічильників; облік боргів (`outstanding`) і
нагадування власнику про прострочені рахунки (Telegram/email); дашборд
`charged/paid/outstanding` + `needs_attention`; базовий графік споживання;
адаптивні media-queries; вкладення договорів; статистика простою; фільтр
статистики за орендарем; імпорт історії з XLSX; курси НБУ USD→UAH.

### #1 — Бекап і відновлення всіх даних (повний знімок)

- **Тип:** ✨ нове · **Хвиля:** 1 · **Вплив:** 🔴 · **Обсяг:** M · **Залежить від:** —
- **Статус:** ✅ уточнено · 🏗️ план: [plans/20260720-backup-and-restore.md](plans/20260720-backup-and-restore.md)
- **Опис:** Кнопка в застосунку: завантажити повний знімок даних і **недеструктивно
  імпортувати** його назад — без доступу до шелу/Synology. Доповнює ручне
  копіювання теки `data/` (яке лишається шляхом для повного відкату).
- **Уточнення:**
  - **Формат:** один архів `.zip` — файл SQLite-бази + вся тека `uploads/` (приватні
    вкладення договорів) + `manifest.json` (версія застосунку, ревізія схеми
    Alembic, час створення, контрольна сума). Без CSV.
  - **Бекап:** консистентний знімок БД через SQLite backup API / `VACUUM INTO` (не
    сире копіювання файла під час запису, бо APScheduler пише в тому ж процесі).
    Завантаження на вимогу зі сторінки *Налаштування*. Планові/автоматичні бекапи —
    поза обсягом (кандидат на потім, поверх наявного scheduler).
  - **Відновлення = недеструктивний імпорт (оновлено 2026-07-20):** додаються лише
    рядки, яких немає, за бізнес-ключами, з ремапом FK; наявні дані не змінюються.
    Уся операція в одній транзакції (атомарно), **без рестарту контейнера**.
    Перевірка сумісності за `manifest.json`: розбіжність ревізії схеми → відмова.
    Раніше планувалась «повна заміна + можливий рестарт» — відхилено: має бути
    недеструктивно й без рестарту (з прицілом на багатьох користувачів).
  - **Модель імпорту:** upsert-missing за ключем; повторний імпорт ідемпотентний
    (0 нових). Вибірковий імпорт і відкат/hot-swap — поза обсягом.
  - **Доступ:** лише автентифікований admin; архів містить приватні дані — не
    логувати вміст, не класти у публічно доступні шляхи.
- **Критерії готовності:**
  - Зі сторінки *Налаштування* можна завантажити `.zip` зі станом БД + `uploads/` +
    `manifest.json`.
  - Імпорт із валідного архіву додає відсутні дані (рахунки, орендарі,
    вкладення-файли) у порожню й у наявну БД, не змінюючи наявних; повторний імпорт
    не створює дублів.
  - Підтвердження обов'язкове; невалідний/несумісний архів відхиляється зі
    зрозумілою помилкою; помилка імпорту → повний rollback (жодних часткових змін).
  - Backend-тести: бекап→відновлення round-trip, відмова на несумісній ревізії
    схеми, захист від неавтентифікованого доступу.
  - `docs/deploy.md` доповнено процедурою (відновлення недеструктивне, без рестарту).

- **Опис:** Облік витрат на квартиру → чистий дохід (оренда − витрати) і маржа.
- **Уточнення:**
  - Нова сутність **«Витрата»**: `apartment_id` (nullable — загальні витрати
    портфеля), `date`, `category` (ремонт/податок/страхування/комісія/інше),
    `amount` + `currency` (дефолт грн), `notes`, опційно чек-вкладення (патерн
    наявних attachments).
  - **Дохід = лише оренда** (`rent_amount_uah` з рахунків); комуналка транзитна й у
    прибуток не входить.
  - **Звіт P&L** за період, по квартирі + загалом: дохід − витрати = чистий; маржа %.
    Суми зводяться в одну валюту через наявні `ExchangeRate` (НБУ).
  - **ROI/дохідність % — поза обсягом v1** (потребує поля «вартість/ціна купівлі»
    квартири); кандидат на потім.
  - **Тренди в часі (поглинуто #10):** у звіті — динаміка витрат і чистого по
    місяцях; тренди споживання можна додати до наявного графіка *Статистики* в межах
    цієї ж роботи. Міжквартирне порівняння — поза обсягом.
- **Критерії готовності:**
  - CRUD витрат: категорія, сума+валюта, дата, прив'язка до квартири (або загальні),
    опційний чек.
  - Звіт показує дохід (оренда), витрати за категоріями, чистий і маржу — по квартирі
    та загалом за обраний період, в одній валюті.
  - Backend-тести: агрегація доходу лише з оренди, підсумки витрат за категоріями,
    конвертація валют, коректний чистий результат.
  - Frontend-тести: додавання/редагування витрати, відображення P&L, зміна періоду.
